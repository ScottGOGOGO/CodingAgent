Vibe Coding Agent — Current Architecture

Current post-refactor architecture. The public API is unchanged, but the internal hot spots are now split into thinner facades and focused modules.

Tier 1 — Playground UI
React + Vite

Composition Root

The UI is intentionally thin and centered on the active chat + preview workflow.

  • apps/playground/src/App.tsx
  • apps/playground/src/main.tsx

Client Hooks

useProject now exposes only the active path: load project, send message, wait for approval, confirm generation, consume SSE.

  • hooks/useProject.ts
  • hooks/useApi.ts

Shared Contract Consumer

The playground still consumes the same shared TS contracts and project events emitted by the API.

  • packages/contracts
Tier 2 — Orchestrator API
Fastify + Runner + Store

HTTP Boundary

Public behavior stays stable across /projects, confirmation, SSE streaming, and /v2.

  • routes/projects.ts
  • routes/v2.ts
  • index.ts wires the composition root

Run Facade and Preflight Coordinator

The former large service files are now thin facades.

  • services/run-service.ts is the run facade
  • services/run/run-session-factory.ts
  • services/run/run-state.ts
  • services/run/run-turn-processor.ts
  • services/run/run-approval-service.ts
  • services/proposal-validator.ts is the preflight coordinator
  • services/preflight/* owns static validation, autofixers, repair context, and manifest defaults

Execution and Persistence

Execution remains outside the agent and stays in the Node orchestration layer.

  • services/execution-worker.ts
  • services/execution-pipeline.ts
  • runner.ts for Docker/local process execution
  • workspace.ts for file ops and preview logs
  • store.ts for memory/Postgres persistence
Tier 3 — Agent Service
FastAPI + LangGraph

Runtime and Strategy

react is still accepted as an input mode, but both modes route through the same PlanSolveStrategy instance.

  • app/main.py
  • app/strategies/plan_solve.py
  • app/strategies/base.py is now a thin adapter

Generation and Verification Services

The verify and guard logic is no longer embedded in the strategy base file.

  • app/services/verify_loop.py owns verify, preflight-repair, and design-polish decisions
  • app/services/generation_guard.py owns placeholder checks, missing local import checks, patch materialization, and critic stub gating
  • clarifier.py, spec_builder.py, context_builder.py, codegen.py, critic.py, model_provider.py

Model Package

The old monolithic model file is now a package barrel while preserving from app.models import ....

  • app/models/common.py
  • app/models/session.py
  • app/models/spec.py
  • app/models/execution.py
  • app/models/structured.py
  • app/models/__init__.py as the compatibility barrel

Main Runtime Flow

User → Playground → Orchestrator /projects / /messages / /confirm / SSE
Orchestrator → RunService facade → RunTurnProcessor / RunApprovalService
Orchestrator → ProposalValidator → preflight/static-validator + preflight/autofixers + agent repair fallback
Orchestrator → Agent Service /agent/turn or /agent/repair
Agent Runtime → PlanSolveStrategy graph → clarify → spec → plan → context → codegen → verify → approval
Approved runs → execution worker → runner + workspace + preview + version persistence

External Dependencies

  • OpenAI-compatible LLM endpoint selected through ModelProvider
  • Optional PostgreSQL via DATABASE_URL
  • Docker or local process runner for install/build/preview execution
  • Workspace filesystem plus Git snapshots

Why This Layout

  • The UI stays thin and event-driven.
  • The Node layer keeps ownership of orchestration, persistence, preview, and execution.
  • The Python layer stays focused on reasoning, generation, critique, and repair.
  • The recent refactor reduces two hot spots: orchestrator run/preflight coordination and agent model/verify concentration.
